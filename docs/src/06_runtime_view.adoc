== Runtime View

This section describes the main runtime collaborations across authentication, profile access, AI matches, realtime online play, rankings and timeout handling.

=== Runtime Scenario 1: Registration and Login

[plantuml, runtime-login, png]
----
actor "User" as Human
participant "React Frontend" as Frontend
participant "Nginx" as Nginx
participant "Auth Service" as Auth
database "authdb" as AuthDB

Human -> Frontend: Fill registration or login form
Frontend -> Nginx: POST /api/auth/register or /api/auth/login
Nginx -> Auth: Forward request
Auth -> AuthDB: create/read credential data
AuthDB --> Auth: stored user / credential row
Auth --> Nginx: access token + refresh token
Nginx --> Frontend: tokens
Frontend --> Human: authenticated session
----

Notable aspects:

* only Auth Service touches credentials,
* token issuance is fully isolated from profile and match persistence,
* Prometheus can observe latency and success rates through auth metrics.

=== Runtime Scenario 2: Authorized Profile Request

[plantuml, runtime-profile-auth, png]
----
actor "User" as Human
participant "React Frontend" as Frontend
participant "Nginx" as Nginx
participant "Users Service" as Users
participant "Auth Service" as Auth
database "users.db" as UsersDB

Human -> Frontend: Open profile page
Frontend -> Nginx: GET /api/users/profiles/:id\nAuthorization: Bearer <token>
Nginx -> Users: Forward request
Users -> Auth: POST /api/auth/verify\nAuthorization: Bearer <token>

alt invalid or unavailable
  Auth --> Users: 401 or upstream failure
  Users --> Nginx: 401 or 503
  Nginx --> Frontend: error response
  Frontend --> Human: redirect or error message
else valid
  Auth --> Users: 200 { valid, claims }
  Users -> UsersDB: SELECT profile
  UsersDB --> Users: profile row
  Users --> Nginx: 200 profile JSON
  Nginx --> Frontend: profile JSON
  Frontend --> Human: render profile
end
----

Notable aspects:

* The auth decision is centralized in Auth Service.
* Short auth timeouts prevent profile requests from hanging indefinitely.

=== Runtime Scenario 3: AI Match Creation and Persistence

[plantuml, runtime-ai-match, png]
----
actor "User" as Human
participant "React Frontend" as Frontend
participant "Nginx" as Nginx
participant "Game Service" as GameService
participant "Auth Service" as Auth
database "gamedb" as GameDB

Human -> Frontend: Create BOT or LOCAL_2P match
Frontend -> Nginx: POST /api/game/matches
Nginx -> GameService: Forward request
GameService -> Auth: POST /api/auth/verify

alt unauthorized
  Auth --> GameService: 401
  GameService --> Frontend: 401
else authorized
  Auth --> GameService: 200 claims
  GameService -> GameDB: INSERT match
  GameDB --> GameService: new match id
  GameService --> Frontend: 201 { matchId }
end
----

Notable aspects:

* Game Service owns durable match creation,
* the browser never talks directly to `gamedb`,
* authorization is checked before any match write happens.

=== Runtime Scenario 4: Human Move Against a Bot

[plantuml, runtime-bot-move, png]
----
actor "User" as Human
participant "React Frontend" as Frontend
participant "Nginx" as Nginx
participant "Gamey" as Gamey
participant "Game Service" as GameService
participant "Auth Service" as Auth
database "gamedb" as GameDB

Human -> Frontend: Play a move
Frontend -> Nginx: POST /api/gamey/v1/ybot/choose/:botId
Nginx -> Gamey: Forward YEN position
Gamey --> Frontend: computed move

Frontend -> Nginx: POST /api/game/matches/:id/moves
Nginx -> GameService: persist move request
GameService -> Auth: POST /api/auth/verify
Auth --> GameService: valid claims
GameService -> GameDB: INSERT move / UPDATE match
GameDB --> GameService: persisted
GameService --> Frontend: 202 processing
----

Notable aspects:

* Gamey remains stateless and performs no persistence,
* browser orchestrates the engine call and then persistence,
* match state is durable only after Game Service accepts the move.

=== Runtime Scenario 5: Rankings and Statistics

[plantuml, runtime-ranking, png]
----
actor "User" as Human
participant "React Frontend" as Frontend
participant "Nginx" as Nginx
participant "Game Service" as GameService
participant "Auth Service" as Auth
database "gamedb" as GameDB

Human -> Frontend: Open rankings or statistics
Frontend -> Nginx: GET /api/game/rankings or /api/game/stats/:userId
Nginx -> GameService: Forward request
GameService -> Auth: POST /api/auth/verify
Auth --> GameService: valid claims
GameService -> GameDB: SELECT ranking/stat rows
GameDB --> GameService: result set
GameService --> Frontend: JSON response
----

Notable aspects:

* rankings and statistics are read from the same durable game store,
* Game Service is the single API for leaderboard and history queries.

=== Runtime Scenario 6: Online Matchmaking and Session Join

[plantuml, runtime-matchmaking, png]
----
actor "Player A" as PlayerA
actor "Player B" as PlayerB
participant "Frontend A" as FrontendA
participant "Frontend B" as FrontendB
participant "Nginx" as Nginx
participant "Game Service / Socket.IO" as GameService
database "Redis" as Redis

PlayerA -> FrontendA: Click Play Online
FrontendA -> Nginx: connect Socket.IO + emit queue:join
Nginx -> GameService: proxy websocket
GameService -> Redis: store queue entry A
GameService --> FrontendA: queue:status

PlayerB -> FrontendB: Click Play Online
FrontendB -> Nginx: connect Socket.IO + emit queue:join
GameService -> Redis: store queue entry B
GameService -> Redis: claim pair and create match assignment
GameService --> FrontendA: matchmaking:matched
GameService --> FrontendB: matchmaking:matched

FrontendA -> GameService: match:join
FrontendB -> GameService: match:join
GameService -> Redis: create/load live session
GameService --> FrontendA: session:state
GameService --> FrontendB: session:state
----

Notable aspects:

* queue state is ephemeral and Redis-backed,
* pairing and live-session creation happen inside Game Service,
* REST polling remains available as a fallback through `/api/game/online/queue/match`.

=== Runtime Scenario 7: Online Move, Chat and Reconnect

[plantuml, runtime-online-session, png]
----
actor "Player A" as PlayerA
actor "Player B" as PlayerB
participant "Frontend A" as FrontendA
participant "Frontend B" as FrontendB
participant "Game Service / Socket.IO" as GameService
database "Redis" as Redis
database "gamedb" as GameDB

PlayerA -> FrontendA: Play move
FrontendA -> GameService: move:play
GameService -> Redis: load session
GameService -> GameService: validate turn and expectedVersion
GameService -> Redis: store updated session
GameService -> GameDB: persist move history
GameService --> FrontendA: session:state
GameService --> FrontendB: session:state

PlayerB -> FrontendB: Send chat message
FrontendB -> GameService: chat:message
GameService -> Redis: append session message
GameService --> FrontendA: chat:message
GameService --> FrontendB: chat:message

PlayerB -> FrontendB: reconnect after disconnect
FrontendB -> GameService: POST /api/game/online/sessions/:matchId/reconnect
GameService -> Redis: clear reconnect deadline
GameService --> FrontendB: updated session snapshot
----

Notable aspects:

* Redis stores the authoritative live session snapshot,
* `gamedb` keeps the durable move history,
* reconnect uses explicit session APIs and grace windows.

=== Runtime Scenario 8: Turn Timeout and Bot Fallback

[plantuml, runtime-timeout, png]
----
participant "OnlineSessionService" as Session
participant "TurnTimerService" as Timer
participant "BotFallbackService" as Fallback
participant "Gamey" as Gamey
database "Redis" as Redis
database "gamedb" as GameDB

Session -> Timer: start timer for active player
Timer --> Session: timeout fired
Session -> Fallback: compute fallback move
Fallback -> Redis: load session
Fallback -> Gamey: POST /api/gamey/v1/ybot/play
Gamey --> Fallback: computed move
Fallback -> Redis: store updated session
Fallback -> GameDB: persist auto move
Session --> Session: emit session:state with bot fallback applied
----

Notable aspects:

* the timeout path reuses the same engine used for normal bot play,
* automatic moves remain auditable because they are persisted,
* reconnect grace is handled before timeout escalation.
